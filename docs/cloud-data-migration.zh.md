# 本地订单迁入云端与安全合并

本说明适用于 `0.1.24` 及以上云端版本。目标是在 Jeff 仍使用离线版期间先准备服务器，等收到最新一致性备份后，将真实订单可核对、可回滚地迁入云端。

## 1. 数据边界

云端应用不能也不应该直接扫描 Jeff 的电脑。Jeff 必须在离线软件的“数据工具”中点击“下载数据库备份”，再由管理员把生成的 `.db` 文件上传到云端迁移工作台。

以下文件不能直接用于正式迁移：

- 单独的 `orders.db-wal`。
- 软件仍在写入时，从安装目录直接复制出来但没有配套 WAL 的 `orders.db`。
- 只有 4 KB、完整性检查后没有订单的空主库。

迁移工作台接受最大 100 MB 的 `.db` 文件。上传只生成预览，不会立即修改云端数据。

## 2. 预检内容

打开云端“数据工具 → 云端数据迁移”，上传备份后系统会检查：

1. SQLite `PRAGMA integrity_check` 必须为 `ok`。
2. 必须存在订单表和订单号字段。
3. 在隔离副本中自动升级旧结构，原上传文件不被修改。
4. 记录原结构版本、文件大小和 SHA-256。
5. 统计订单、交货记录、操作历史及各状态订单数量。
6. 按“公司 + 订单号”对比本地和云端数据。
7. 识别这份文件是否已经迁移过，阻止重复执行。

预览有效期为 24 小时。正式执行前还会重新计算云端指纹；如果预览后有人新增、修改或核销了云端订单，本次操作会停止并要求重新上传预览。

## 3. 安全合并规则

完成第一次迁移后，系统会为每张订单保存本地与云端基线。后续上传新备份时按以下规则处理：

| 差异 | 默认处理 |
| --- | --- |
| 仅本地备份中有 | 新增到云端 |
| 仅云端中有 | 保留云端 |
| 两边完全相同 | 保留现状 |
| 基线后只有本地修改 | 采用本地 |
| 基线后只有云端修改 | 保留云端 |
| 本地和云端都修改 | 必须逐单选择“采用本地”或“保留云端” |
| 上次已保留的历史差异且两边都没再改 | 继续保留云端 |

采用本地订单时会连同稳定 ID、创建和更新时间、先交要求、实际交货记录、退回信息及操作历史一起迁入。整个合并在数据库事务中执行。

## 4. 两种迁移方式

### 完整替换

适合首次正式上线，且最新离线备份是唯一准确信息源、云端还没有需要保留的真实订单时使用。它会整体替换云端业务数据库，但保留云端登录密钥和管理员密码设置。

当上传文件没有订单、云端已有订单时，系统会禁止完整替换。

### 安全合并

适合以下情况：

- 云端试用库已有需要保留的订单。
- Jeff 提供过一次备份后仍继续使用离线版。
- 云端和离线端在同一时期都发生过真实修改。
- 第一次迁移后还要补入最终切换前的新订单。

发生真正冲突时不会按更新时间猜测，必须由管理员逐单确认。

## 5. 执行与回滚保护

执行迁移时系统会：

1. 创建维护锁，暂时拒绝新增、修改、交货、退回、核销和普通导入操作。
2. 使用 SQLite Backup API 生成 `before-cloud-migration` 一致性备份。
3. 再次确认云端数据与预览时一致。
4. 完成替换或事务合并。
5. 执行迁移后完整性检查。
6. 写入迁移基线、文件哈希和结果报告。
7. 任一步失败时自动恢复迁移前备份。

迁移页面会显示最近的迁移报告。只允许回滚当前数据库最近一次迁移；回滚前还会生成 `before-migration-rollback` 安全备份，恢复后再次检查完整性。回滚需要输入“回滚迁移”确认。

每日备份和这些迁移备份仍应同步到另一台机器或对象存储，避免服务器磁盘本身损坏。

## 6. Jeff 正式切换步骤

1. 服务器先升级到最新云端版本并通过 `/api/health`。
2. Jeff 可继续用离线版，服务器准备过程不需要提前停工。
3. 约定最终切换时间后，Jeff 暂停在离线版登记、补交、退回和核销。
4. Jeff 在离线软件“数据工具”点击“下载数据库备份”。
5. 管理员把该 `.db` 上传到云端迁移工作台并核对文件名、订单数、状态数和完整性。
6. 云端没有真实订单时选择“完整替换”；已有试用或真实数据时选择“安全合并”。
7. 对所有冲突订单逐项确认后执行。
8. 核对迁移报告，并随机抽查待核销、部分交货、已退回、已核销和有多次先交记录的订单。
9. Jeff 与助理分别用手机和电脑完成一次查询、新增和核销验收。
10. 确认后把云端设为唯一日常数据源；离线版只保留为只读历史备份，不再继续录入。

如果 Jeff 在第一次备份后又产生了离线修改，不必手工抄写。取得最终新备份后再次上传，系统会基于上次迁移基线识别单边修改和冲突。
